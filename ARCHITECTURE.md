# PokerFace — техническая архитектура

> Онлайн-игра «не улыбнись». Игроки заходят в лобби с веб-камерами (как в Discord),
> начинается матч. Улыбнулся → жёлтая карточка. Улыбнулся второй раз → красная,
> вылет. Побеждает последний, кто сохранил «железное лицо».
>
> Платформы: iOS + Android (один код, React Native + Expo).
> Видео: LiveKit Cloud. Игровой сервер: Colyseus (Node.js).
> Детект улыбки: на устройстве (ML Kit / MediaPipe), сервер не видит видео.

---

## 1. Принципы архитектуры

1. **Детект улыбки — только на устройстве.** Сервер никогда не анализирует видео.
   Каждый телефон сам смотрит на свою камеру и шлёт серверу лишь событие
   «я улыбнулся» с уверенностью и таймстампом. Это даёт: приватность, нулевую
   нагрузку на сервер от ML, мгновенную реакцию, дешёвый масштаб.
2. **Сервер — авторитарный судья.** Клиент не решает, выбыл ли игрок. Клиент только
   репортит «улыбку». Карточки, вылеты и победителя считает сервер. Это защищает
   от читов (нельзя подделать «я не улыбался»).
3. **Три независимых слоя.** Видео (LiveKit), игровая логика (Colyseus),
   детект улыбки (on-device) развязаны. Любой можно менять отдельно.
4. **Anti-cheat с самого начала.** Камера обязана быть включена и показывать лицо.
   Нет лица в кадре N секунд → авто-штраф. Это часть правил, не доп. фича.

## 2. Стек

| Слой | Технология | Почему |
|------|-----------|--------|
| Мобильное приложение | React Native + Expo (dev client) | один код на iOS/Android, быстрый цикл |
| Язык | TypeScript | общие типы между клиентом и сервером |
| Камера + ML на кадрах | react-native-vision-camera + frame processors | доступ к кадрам в реальном времени |
| Детект улыбки | ML Kit Face Detection (MVP) → MediaPipe Face Landmarker (точность) | `smilingProbability` из коробки |
| Видеосвязь | @livekit/react-native (LiveKit Cloud) | SFU «как Discord», managed |
| Игровой сервер | Colyseus (Node.js) | авторитарные комнаты, синхронизация состояния |
| Realtime-транспорт игры | WebSocket (встроен в Colyseus) | события улыбок, карточки, фазы |
| БД | PostgreSQL (профили, рейтинг) + Redis (сессии/матчмейкинг) | стандарт |
| ИИ-ведущий (фича) | Claude API (claude-opus-4-8 / claude-haiku-4-5) | генерация подколов |
| Авторизация | Clerk или Supabase Auth | быстрый старт, соц-логины |
| Сборка/релиз | EAS Build + EAS Submit | сборка в облаке, выкладка в сторы |
| Бэкенд-хостинг | Railway / Render / Fly.io | простой деплой Node + Postgres |

## 4. Модель данных

### 4.1 Синхронизируемое состояние комнаты (Colyseus schema)

Это то, что сервер автоматически реплицирует всем клиентам в комнате.

```ts
// server/game/src/schema/GameState.ts
class Player extends Schema {
  @type("string") id: string;          // sessionId
  @type("string") userId: string;      // постоянный id профиля
  @type("string") name: string;
  @type("string") avatarUrl: string;
  @type("string") livekitIdentity: string; // связь с видео-дорожкой
  @type("uint8")  cards: number = 0;        // 0 = чисто, 1 = жёлтая, 2 = красная(вылет)
  @type("boolean") eliminated: boolean = false;
  @type("boolean") faceVisible: boolean = true;
  @type("boolean") ready: boolean = false;
  @type("uint32") survivedMs: number = 0;   // для рейтинга
}

class GameState extends Schema {
  @type("string") phase: string = "lobby";  // см. конечный автомат ниже
  @type("string") hostId: string;
  @type({ map: Player }) players = new MapSchema<Player>();
  @type("uint32") roundStartedAt: number = 0;
  @type("string") winnerId: string = "";
  @type("string") tauntText: string = "";   // текущая фраза ИИ-ведущего
}
```

### 4.2 Постоянное хранилище (Postgres) — упрощённо

```
users         (id, handle, display_name, avatar_url, created_at)
stats         (user_id, elo, games_played, wins, longest_survival_ms, iron_face_streak)
matches       (id, started_at, ended_at, winner_id, mode)
match_players (match_id, user_id, placement, cards, survived_ms)
clips         (id, user_id, match_id, video_url, created_at)  -- авто-клипы «провала»
```

## 5. Протокол событий (клиент ↔ сервер)

Общие константы в `packages/shared`.

### Клиент → Сервер

| Сообщение | Payload | Когда |
|-----------|---------|-------|
| `join_lobby` | `{ code }` | вход в лобби по коду |
| `set_ready` | `{ ready }` | готовность |
| `start_game` | — | только host |
| `smile_detected` | `{ confidence, ts }` | детектор поймал улыбку (с антидребезгом) |
| `face_lost` / `face_found` | `{ ts }` | лицо пропало/вернулось в кадр |
| `use_item` | `{ itemId, targetId }` | применить «снаряд» (фича Провокатор) |

### Сервер → Клиент (поверх авто-синка состояния)

| Сообщение | Payload | Смысл |
|-----------|---------|-------|
| `card_issued` | `{ playerId, cards }` | выдана жёлтая/красная |
| `player_eliminated` | `{ playerId }` | вылет (триггерит авто-клип) |
| `phase_changed` | `{ phase }` | смена фазы |
| `taunt` | `{ text, targetId }` | реплика ИИ-ведущего |
| `game_over` | `{ winnerId }` | конец матча |

> **Важно про честность:** `smile_detected` — это сигнал, а не приговор. Сервер
> сам решает, штрафовать ли (дебаунс, кулдаун между карточками, защита от спама).

## 6. Конечный автомат матча

```
        ┌────────┐  все ready & host start   ┌─────────────┐
        │ LOBBY  │ ─────────────────────────►│  COUNTDOWN  │  (3..2..1)
        └────────┘                           └──────┬──────┘
             ▲                                       │
             │ rematch                               ▼
             │                                ┌─────────────┐
        ┌────┴─────┐  остался 1 игрок         │   PLAYING   │◄─┐
        │ GAME_OVER│◄─────────────────────────│             │  │ улыбки,
        └──────────┘                          └──────┬──────┘  │ карточки,
                                                     │         │ вылеты
                                                     └─────────┘
```

- **LOBBY** — собираются игроки, видео уже идёт, можно «прогреваться».
- **COUNTDOWN** — 3 секунды, камеры всех на весь экран, нагнетание.
- **PLAYING** — детект активен. Улыбка → `cards++`. При `cards == 2` → вылет.
  Сервер периодически дёргает Claude за `taunt`.
- **GAME_OVER** — победитель, статистика, кнопка «реванш».

Правила-параметры (в `packages/shared/config.ts`, легко крутить для баланса):
```ts
SMILE_THRESHOLD = 0.7        // порог вероятности улыбки
SMILE_FRAMES_TO_TRIGGER = 3  // сколько кадров подряд выше порога
CARD_COOLDOWN_MS = 2500      // нельзя получить 2 карточки за один смех
FACE_LOST_GRACE_MS = 4000    // сколько можно прятать лицо до штрафа
COUNTDOWN_MS = 3000
```

## 7. Детект улыбки (сердце игры)

Поток на устройстве:

```
Камера (front) ──► VisionCamera frame processor
                      │  (10–15 кадров/сек, не каждый — экономим батарею)
                      ▼
              ML Kit Face Detection
                      │  smilingProbability ∈ [0,1]
                      ▼
        Антидребезг (скользящее окно):
        N кадров подряд > SMILE_THRESHOLD?
                      │ да
                      ▼
        Отправить серверу smile_detected
        (один раз, потом кулдаун)
```

Почему так:
- **Окно из N кадров** убирает ложные срабатывания (моргнул, повернулся).
- **Кулдаун** — чтобы один приступ смеха не дал сразу две карточки.
- **MVP — ML Kit** (`smilingProbability`). Позже **MediaPipe Face Landmarker**
  даёт blendshapes `mouthSmileLeft/Right` + можно ловить «глаза смеются»
  (`cheekSquint`) → отличать вежливую ухмылку от настоящего смеха.

Edge-кейсы (закладываем сразу):
- Нет лица в кадре → `face_lost` → через grace-период штраф (анти-чит).
- Несколько лиц в кадре → подозрение на чит → флаг хосту.
- Тёмное помещение → подсказка «добавь света» (качество детекта).

## 8. Виральная петля — авто-клип «момент провала» (killer-фича)

Это то, что даст органический рост. Механика:

1. Клиент **постоянно** держит кольцевой буфер последних ~6 секунд кадров своей
   камеры локально (только в памяти).
2. Когда сервер шлёт `player_eliminated` для этого игрока — клиент берёт буфер,
   делает 5-сек **замедленный** ролик момента, накладывает оверлей
   («🟥 ВЫЛЕТЕЛ на 02:14 • PokerFace») и предлагает «Сохранить / Поделиться».
3. Шаринг в TikTok/Reels/Shorts с водяным знаком и ссылкой → новые игроки.

> Приватность: клип делается из **своей** камеры, локально, и публикуется только
> по явному тапу игрока. Чужие лица в клип не попадают (MVP).

## 10. Поэтапный план разработки

**Этап 0 — каркас (1 спринт)**
- Монорепо, общие типы, Colyseus `GameRoom` со схемой, Expo-приложение,
  экраны Home/Lobby/Game-заглушки, локальный матч из 2 устройств.

**Этап 1 — ядро игры (риск №1)**
- VisionCamera + ML Kit, детект улыбки с антидребезгом, событие на сервер,
  логика карточек/вылетов, конечный автомат до победителя. *Без видео пока.*

**Этап 2 — видео**
- LiveKit Cloud: токен-сервис, публикация/подписка дорожек, сетка камер в лобби
  и в игре. Связка `livekitIdentity` ↔ игрок.

**Этап 3 — петля роста**
- Кольцевой буфер кадров, генерация и шеринг авто-клипа.

**Этап 4 — мета**
- Авторизация, профили, ELO-рейтинг, матчмейкинг, лиги.

**Этап 5 — фишки**
- ИИ-ведущий, арсенал «провокатора», зрители/ставки, AR.

**Этап 6 — релиз**
- EAS Build, тест на реальных устройствах, App Store + Google Play.

## 11. Как запустить и опубликовать

1. **Dev:** Expo dev client + `colyseus` локально + LiveKit Cloud (free tier).
   Тест на 2 физических телефонах (камера/ML на эмуляторе работают плохо).
2. **Аккаунты:** Apple Developer ($99/год), Google Play Console ($25 разово).
3. **Сборка:** `eas build -p ios` / `-p android`.
4. **Выкладка:** `eas submit`. TestFlight (iOS) и внутреннее тестирование (Play)
   для беты до публичного релиза.
5. **Бэкенд:** Colyseus + Postgres + Redis на Railway/Fly.io. LiveKit Cloud отдельно.

## 13. Главные риски и как их снимаем

1. **Точность детекта улыбки** → начинаем именно с этого (Этап 1), тюним порог
   и окно на реальных лицах; путь апгрейда на MediaPipe заложен.
2. **Задержка/качество видео при 6+ игроках** → LiveKit SFU тянет это; ML локально,
   не зависит от видеоканала.
3. **Читы (отвернулся/закрыл камеру)** → обязательная камера + анти-чит правила
   с первого дня (раздел 7).
4. **Батарея/нагрев от ML на каждом кадре** → обрабатываем не каждый кадр (10–15 fps),
   останавливаем детект вне фазы PLAYING.
5. **Рост** → виральный авто-клип (раздел 8) встроен как механика, а не «потом».
